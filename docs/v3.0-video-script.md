## SECTION 1: Agent-Driven Installation (0:30 – 1:45)

**[Screen: release page — Agent-Driven Installation section]**

> The first thing you'll see is agent-driven installation,
> and it says breaking change.
> That's because the old Makefile is gone.
>
> Before this, every time I updated a command,
> I had to run `make install`
> and then go project by project to copy files over.
> I work on over 10 projects,
> so I kept losing track of which version was where.
>
> Now you just clone the repo
> and tell your AI agent
> "read CLAUDE.md and install what I need."
>
> The agent figures out your project context,
> checks what MCP servers you have,
> looks at what's already installed,
> and syncs everything in place.
>
> Updates work the same way —
> pull the latest, tell the agent to check again,
> it finds what changed and updates just those files.
> No shell scripts to maintain.
> The agent is the installer.

---

## SECTION 2: Two-Tier Command Architecture (1:45 – 3:00)

**[Screen: release page — Two-Tier Command Architecture section]**

> The second big change is how commands are organized.
>
> Claude Code gives you two places to put custom commands.
> Your personal folder at `~/.claude/commands/` —
> anything there is available in every project you open.
> And the project folder at `.claude/commands/` —
> those only show up in that one project.
>
> Before version 3,
> I had the same commands duplicated everywhere.
> Six copies of "evolve." Six copies of "dw-plan."
> Fix a bug in one, the other five stay broken.
>
> Now the toolkit tells you which commands go where.
> Universal stuff like "evolve," "session-summary,"
> "compare," "sync," and the dev workflow commands —
> those go in your personal folder.
> Install once, use everywhere.
>
> Commands that depend on specific MCP servers —
> like Jira, TestLink, or Confluence —
> those stay in the project
> because not every project needs them.
>
> And there's a command called "review-install"
> that audits your setup, catches duplicates,
> and tells you if anything is in the wrong place.

## SECTION 3: New Commands (3:00 – 3:45)

**[Screen: release page — New Commands table]**

> Version 3 also adds a bunch of new commands.
> You can see the table right here.
>
> The one I want to highlight is "evolve."
> It analyzes your command usage patterns
> and suggests improvements.
>
> It builds on top of "session-summary,"
> which records what happened in each conversation
> without saving any credentials or private data.
> The more sessions you record,
> the better "evolve" gets at finding problems.

## SECTION 4: Dev Workflow (3:45 – 4:45)

**[Screen: README — Issue-Driven Development section]**

> Now let me switch over to the README
> and show you the dev workflow.
>
> I actually built this entire release
> using its own dev workflow commands.
> You can see the flow right here.
>
> "dw-plan" breaks your request into GitHub issues.
> "dw-implement" picks up an issue,
> creates a branch, and starts coding.
> "dw-create-pr" opens a pull request linked to that issue.
> Then "dw-review-pr" and "dw-merge" to finish it off.
> Every change tied back to a GitHub issue, start to finish.
